Overview

Recursive Language Models (RLMs), developed at MIT CSAIL, enable any LLM to process 10M+ tokens without retraining. An RLM is a thin wrapper around a language model: from the user's perspective, rlm.completion(messages) is a drop-in replacement for gpt5.completion(messages). Under the hood, the prompt lives as data in an external Python REPL and the model writes code to peek, chunk, grep, and recursively call itself on smaller slices.

Unlike agents that decompose by task, RLMs decompose by context - the LM decides how to break down its input. RLM(GPT-5-mini) outperforms GPT-5 by +114% on OOLONG at comparable cost. RLM-Qwen3-8B, a post-trained open model, achieves +28.3% median gain over its base.

Motivation
  • Context rot is real. As conversation length grows, models get measurably worse - even within their context window. On OOLONG, GPT-5 scores drop as context doubles from 132K to 263K tokens.
  • Summarization heuristics assume early context is forgettable - wrong for tasks needing Ω(|P|) work (e.g. counting, labeling thousands of entries).
  • No symbolic handle: the prompt exists only as tokens inside the model - no way to index, slice, grep, or loop over it programmatically.
  • Verbal sub-delegation: prior agent scaffolds verbalize a few sub-tasks; they cannot launch Ω(|P|) programmatic sub-calls in a tight loop.
  • Output bounded by K: direct generation cannot produce responses longer than the context window.
Core Idea

An RLM makes three design choices missing from existing scaffolds:

  1. Prompt as environment data. The full prompt P lives outside the transformer as a variable in a persistent Python REPL.
  2. Constant-size root context. The root language model sees only short metadata (length, prefix, stdout summaries) - never the raw prompt tokens.
  3. Symbolic recursion. The model writes Python that peeks, chunks, searches, and calls sub_RLM(…) inside loops. When Final is set in the REPL, iteration stops and its value is returned.
Architecture & Methods

Given prompt P, an RLM initialises a Python REPL with P as a variable and a sub_RLM() hook. The loop: model generates code → REPL executes → constant-size metadata returns → repeat until Final is set.

Where PyTorch Fits in the Stack

The recursive control flow is programmatic (Python), not neural. PyTorch powers the language model - every root step and every sub-call is one autoregressive generate() pass on GPU.

System stack - where PyTorch sits
REPL
Persistent Python env. Stores P as a variable, intermediate chunks, sub_RLM() hook, stdout buffer.
Orchestrator
RLM loop: assemble hist (constant-size metadata only) → call the language model → execute returned code → trim stdout → repeat.
PyTorch
transformers / vLLM / TGI load Qwen3-8B or RLM-Qwen3-8B weights. Each root step & each sub-call = one model.generate() on H100 GPU.
Training
SFT via prime-rl (PyTorch): distill 1K filtered trajectories from Qwen3-Coder-480B RLM → RLM-Qwen3-8B. ~48 H100-hours.

Key distinction: recursion = program control flow in the REPL, not extra transformer layers. PyTorch handles the language model’s inference and the SFT training loop.

Emergent RLM Strategies

Without explicit training, RLMs discover reusable patterns for handling long context:

StrategyWhat the Root LM DoesWhen Used
PeekingReads first N chars to discover structureStart of every RLM call
GreppingRegex / keyword search to narrow linesNeedle-in-haystack, filtering
Partition + MapChunks context, sub_RLM() per chunkDense labeling, counting
SummarizeCondenses sub-call outputs to aggregateMulti-hop reasoning
ProgrammaticWrites code to process diffs / computeLong-output tasks

These strategies compose: peek first, grep to filter, then partition + map sub-calls over filtered slices.
